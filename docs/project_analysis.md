# Open Harness プロジェクト分析レポート

## 概要

Open Harness はローカルLLM向けの自律エージェントハーネス。弱いモデルでも安定して動作する補償機能、外部エージェント委譲、Git ベースのチェックポイント、ポリシーベースの安全機構を備える。

---

## 強み (Strengths)

### S1. ローカルLLMファーストという明確なポジショニング
- Ollama / LM Studio など OpenAI 互換 API を前提とした設計
- プライバシー重視の組織・個人開発者に刺さる独自のニッチ
- クラウドAPI 依存のツール (Claude Code, Cursor, Cline 等) との差別化が明確

### S2. Weak Model Compensation（弱モデル補償）
- 他ツールにない独自機能。エラー分類→修復→エスカレーションの3段パイプライン
- JSON 修復、ファジーツール名マッチング、prose-wrapped 応答のアンラップ
- 小型モデルでも実用レベルの安定動作を実現

### S3. 外部エージェントオーケストレーション
- Claude Code / Codex / Gemini CLI を子エージェントとして委譲
- ローカルLLMが「指揮者」、クラウドエージェントが「演奏者」という独自アーキテクチャ
- 小型モデルの弱点をクラウドエージェントで補完できる

### S4. 安全機構の充実
- 3層のポリシープリセット (safe / balanced / full)
- パス ACL、シェルコマンドブロッカー、承認エンジン、サンドボックス
- Git ベースチェックポイントによるロールバック
- 企業導入に向けた信頼性の基盤あり

### S5. 堅実なアーキテクチャ (v2.0)
- async-first 設計、EventBus によるUI疎結合
- 4層コンテキスト管理、適応的トークン圧縮 (40-60% 削減)
- 複雑度適応プランニング (Low/Medium/High)
- ~8,000行と過剰でないコードベース

### S6. テスト・CI/CD 基盤
- 29個のテストファイル、5,468行のテストコード
- GitHub Actions で pytest + ruff を自動実行
- integration テストのマーカー分離

### S7. 日英バイリンガルドキュメント
- README、チュートリアル、外部エージェントガイドが日英両対応
- 日本語圏の開発者に対するアクセシビリティ

---

## 弱み (Weaknesses)

### W1. エコシステム・プラグイン拡張性の不足
- ツールは 15 個の組み込みのみ。MCP (Model Context Protocol) 非対応
- サードパーティプラグインの仕組みがない
- **競合比較**: Claude Code は MCP サーバー経由で無限に拡張可能。Cline も MCP 対応済み

### W2. マルチモーダル対応の欠如
- テキストのみ。画像・スクリーンショット・PDF の入力非対応
- **競合比較**: Claude Code はマルチモーダル入力対応、画面キャプチャも可能

### W3. ストリーミング・UX の洗練度
- TUI はオプション扱いで基本はシンプルな REPL
- ストリーミング出力のリッチさが競合に劣る
- **競合比較**: Claude Code / Cursor はリアルタイムストリーミング＋構文ハイライト付き

### W4. Python 3.14+ 要件の高さ
- `requires-python = ">=3.14"` は現時点で非常に限定的
- 多くの開発者が Python 3.10-3.12 を使用しており、導入障壁が高い
- **影響**: 潜在ユーザーの大部分を除外してしまう

### W5. コンテキストウィンドウの制約
- ローカル LLM は典型的に 4K-32K トークン (クラウドの 200K と比較)
- 大規模コードベースでの長期タスク実行が困難
- 圧縮機構はあるが根本的なトークン制限は残る

### W6. ベンチマーク・実績データの不足
- SWE-bench 等の標準ベンチマークでの評価なし
- 「弱モデル補償で X% の成功率向上」等の定量データがない
- 信頼性を主張するための裏付けが弱い

### W7. コミュニティ・エコシステムの未成熟
- GitHub スター数やコントリビューター数が不明瞭
- プラグイン・拡張のエコシステムがない
- ユーザー事例やドキュメントの外部リソースが少ない

### W8. バージョン 2.0.0a3（アルファ）
- まだプレリリース段階。API の安定性が保証されない
- v1 テストがレガシーとして残存（移行未完了）
- プロダクション導入の判断材料としては弱い

## 戦略的推奨: 優先実装すべき機能

### 優先度 High（競争力に直結）

1. **MCP (Model Context Protocol) サーバー対応**
   - 理由: エコシステム拡張性の最大のギャップ。MCP対応でツール数が飛躍的に増加
   - 効果: Claude Code / Cline ユーザーの移行障壁を下げる
   - 工数: 中〜大

2. **Python バージョン要件の緩和 (3.10+)**
   - 理由: 現状で潜在ユーザーの大部分を除外している
   - 効果: 即座に対象ユーザー層が 10 倍以上に拡大
   - 工数: 小〜中

3. **マルチモーダル入力対応（画像/スクリーンショット）**
   - 理由: UI バグ修正等のワークフローで必須化しつつある
   - 効果: ユースケースの大幅拡大
   - 工数: 中

### 優先度 Medium（差別化の強化）

4. **ベンチマーク & 実績の定量化**
   - SWE-bench lite でのモデル別成功率を計測・公開
   - 弱モデル補償あり/なしの比較データ

5. **RAG (Retrieval-Augmented Generation) 統合**
   - コードベース全体のインデックス作成
   - ローカルLLMのコンテキスト制限を実質的に緩和

6. **ストリーミング UX の改善**
   - 構文ハイライト付きストリーミング
   - diff プレビュー、ファイル変更のリアルタイム表示

### 優先度 Low（段階的改善）

7. **IDE 統合 (VS Code 拡張)**
8. **Docker/コンテナベースのサンドボックス**
9. **Web UI ダッシュボード**
10. **プラグインマーケットプレイス**

---

## まとめ

Open Harness の最大の強みは「ローカルLLMファースト」「弱モデル補償」「外部エージェントオーケストレーション」という、他のどの競合も持たない独自の組み合わせにある。一方、最大の弱みは「拡張性の不足（MCP非対応）」「Python 3.14+要件」「マルチモーダル非対応」である。

短期的には **Python バージョン要件の緩和** と **MCP 対応** が、ユーザー獲得に最もインパクトが大きい施策となる。
